# 21. 탐험 버튼 단순화 + 짧은 시간 효율 우대 복원안

> **상태**: 적용 완료. 본 문서는 변경 이력/근거 기록용.
> **관련 코드**: `src/lib/game/data.ts` (`DISPATCH_DURATIONS`, `DISPATCH_REWARD_MULT`), `src/lib/game/logic.ts` (`resolveDispatch` durationMult 적용부, log cap), `src/app/tabs/ExploreTab.tsx` (버튼 렌더 + 툴팁), `src/lib/game/store.ts` (`startDispatch`, persist 마이그레이션)
> **연관 문서**: `docs/05-content.md` (탐험 표), `docs/08-balance-reference.md` (Dispatch 효율 표), `docs/18-cancel-partial-reward-plan.md` (전제 변경 — §6 참고)

## 1. 문제 정의

현재 지역 카드마다 탐험 시간 버튼이 4개(60s / 1800s / 7200s / 28800s) 노출된다. 지역 16개 × 4 버튼 = 한 화면에 64개 클릭 타깃.

```ts
// src/lib/game/data.ts:2184-2193
export const DISPATCH_DURATIONS = [60, 1800, 7200, 28800] as const;
export const DISPATCH_REWARD_MULT: Record<DispatchDuration, number> = {
  60: 1.0,
  1800: 1.0,
  7200: 1.0,
  28800: 1.0,
};
```

문제 두 가지를 묶어서 해결한다.

- **UI 과밀**: 버튼 4개 중 **2시간 ↔ 8시간**의 효율이 동등하다는 정보를 매번 툴팁으로 알려야 하는데(둘 다 100%), 그 메시지를 강조할수록 "그럼 왜 8시간을 따로 두지?"라는 의문이 커진다.
- **선택의 무의미함**: 길이별 효율을 1.0으로 통일한 직전 변경(`8f632de`)으로 시간 선택이 "내가 자리에 얼마나 없을 것인가"의 일정 관리 도구가 되었다. 그러나 방치형 게임의 클래식한 즐거움 — **활성 플레이 시 짧게 자주 돌리며 보상이 더 많이 쌓이는 손맛** — 이 사라졌다.

## 2. 목표

1. 탐험 버튼을 **3개로 축소**: 60s(1분) / 1800s(30분) / 7200s(2시간). 8시간 옵션 제거.
2. **짧은 시간 = 높은 보상 효율** 곡선 복원. (직전 commit `8f632de` 이전 의도였던 동등 효율 폐기 + 더 이전 commit `db6c4b4`의 곡선 부활 + 새 길이 스케일에 맞춰 재조정)
3. 변경에 따라 영향받는 표/문서/마이그레이션 / 보스 외 진행 코드 일괄 동기화.

## 3. 보상 효율 곡선 — 후보 비교

### 참고: 직전 (commit `db6c4b4`, 길이 10/30/60/300s 시절)

| 시간 | 효율  | 효율/초 |
| ---- | ----- | ------- |
| 10s  | 1.00x | 1.00    |
| 30s  | 0.95x | 0.95    |
| 60s  | 0.85x | 0.85    |
| 300s | 0.70x | 0.70    |

→ "활성 플레이 페널티 없음, 자리 비울수록 효율 30%까지 손해" 라는 부드러운 곡선.

### 3.A 동등 페널티 폭 유지 (보수적)

길이가 30배·120배 점프하는 새 스케일에 동일한 30% 폭을 그대로 매핑.

| 시간              | 효율  | 효율/초 | 1시간 환산 보상 |
| ----------------- | ----- | ------- | --------------- |
| 60s (1분)         | 1.00x | 1.00    | 3,600 (60×60)   |
| 1800s (30분)      | 0.85x | 0.85    | 3,060 (60×51)   |
| 7200s (2시간) ÷ 2 | 0.70x | 0.70    | 2,520 (60×42)   |

활성 1분 반복(60회) ↔ 30분 방치(2회) ↔ 2시간 방치(0.5회) → **3600 / 3060 / 2520**. 활성-방치 격차 약 1.43배.

### 3.B 중간 격차 (방치형 정체성과 활성 손맛의 균형)

활성 vs 방치 격차가 너무 크면 "1분 클릭 매크로가 정답"이 되고, 너무 작으면 §1의 "왜 시간을 고르지?" 문제가 재발.

| 시간          | 효율  | 효율/초 | 1시간 환산 보상 |
| ------------- | ----- | ------- | --------------- |
| 60s (1분)     | 1.00x | 1.000   | 3,600           |
| 1800s (30분)  | 0.88x | 0.880   | 3,168           |
| 7200s (2시간) | 0.75x | 0.750   | 2,700           |

활성-방치 격차 약 1.33배. **추천안 — 본 문서 기준.** 1차 0.92/0.82안에서 방치 페널티를 한 단계 더 강화한 값(30분 -4%p, 2시간 -7%p) — 길이 점프(30x → 120x)가 큰 만큼 2시간 쪽을 더 깎아 곡선의 우하향 기울기를 분명히 함.

### 3.C 활성 보너스만 부여, 방치 페널티 없음

| 시간          | 효율  |
| ------------- | ----- |
| 60s (1분)     | 1.10x |
| 1800s (30분)  | 1.00x |
| 7200s (2시간) | 1.00x |

심리적으로는 매력적이나, 1분 버튼만이 "특별 효율"이 되어 다른 두 버튼이 다시 동등해짐 → §1의 문제가 재발. 기각.

→ **3.B 채택**. §6에서 사용자 결정 한 번 받고 확정.

### 4.2 보상 적용부 (`src/lib/game/logic.ts:1264-1270`)

이미 `DISPATCH_REWARD_MULT[durationSec] ?? 1.0` 형태로 안전 fallback이 들어 있다(주석 "구버전 durationSec(10/30/300 등) 잔존 안전장치"). 8시간(28800)·구버전 값이 실행 중인 세이브에 남아 있어도 `1.0`으로 처리되어 **보상 손실은 없다**. 코드 변경 불필요.

```ts
// 기존 그대로 유지
const durationMult = DISPATCH_REWARD_MULT[durationSec] ?? 1.0;
```

검토 포인트: 28800 케이스에 한해서는 fallback이 의도와 약간 다르게 풀린다(폐기되었지만 진행 중인 경우 100% 그대로 지급). 이는 **재시작 시점에 진행 중이던 한 번의 탐험에만** 적용되며, 다음 출발부터는 7200이 최대값이라 자연 소거.

### 4.3 로그 캡 (`src/lib/game/logic.ts:1276-1277`)

8시간(28,800턴) 방어용 `LOG_CAP = 500`은 **그대로 유지**.

- 7200턴까지는 캡 없이도 메모리/저장소 부담이 작다고 볼 수도 있으나, 캡을 낮추는 변경은 **이번 작업 범위 밖**으로 미룬다. (별도 PR로 평가 — 100~200 정도가 UX상 적절한지 측정 필요.)
- 캡 자체는 leaky하지 않게 두는 편이 안전 (회귀 시 28800이 다시 살아날 가능성 0이 아님).

### 4.4 ExploreTab UI (`src/app/tabs/ExploreTab.tsx:144-157`)

코드 변경은 사실상 없음 — `DISPATCH_DURATIONS.map`이 자동으로 3개를 그린다. 효율 툴팁(`DISPATCH_REWARD_MULT[d] * 100`)도 자동 동작.

확인할 것:

- 버튼 폭이 좁아져 한 줄에 깔끔히 들어가는지 (현재 `flex-wrap`이라 OK).
- 효율 툴팁이 1분에서만 100%를 보여주므로 사용자가 차이를 인지할 수 있는지 — 추가로 60s 버튼에 `+` 강조 같은 시각 단서가 있으면 좋다. **첫 패치에는 미포함**, 반응 보고 결정.

### 4.5 Store / persist (`src/lib/game/store.ts:79`, `:501-553`)

- `startDispatch(regionId, durationSec: DispatchDuration)` 시그니처는 `DISPATCH_DURATIONS` 타입에서 자동 추론되므로 28800 호출 경로가 컴파일 타임에 차단됨.
- **persist 마이그레이션**: 사용자의 진행 중 탐험이 28800일 가능성. `s.dispatch.durationSec === 28800`이면 `cancelDispatch` + alert 또는 강제 `finalizeDispatch`. 단순화를 위해 **그대로 두고 끝나면 정산** (§4.2 fallback 1.0 적용)을 채택. 마이그레이션 코드 불필요.
- 단, **store load 시 잔존 28800 dispatch가 있으면** 로드 후 그대로 진행. progress bar 등 UI는 28800 기준으로 정상 표시됨 (durationSec 필드 자체는 보존). 새 탐험을 시작하는 순간부터 3개 버튼만 노출.

## 5. UX 영향 점검

| 시나리오                  | 현재                   | 변경 후                                            |
| ------------------------- | ---------------------- | -------------------------------------------------- |
| 출근 전 8시간 출발        | 한 번에 8시간 ✓        | 2시간 × N 또는 자기 전 다른 게임 — **불편 가능성** |
| 점심시간 30분 자리비움    | 30분 1회               | 30분 1회 (변동 없음, 효율만 88%로 하락)            |
| 활성 플레이 중 1분씩 반복 | 60회 × 1.0 = 동일 효율 | 60회 × 1.0 = **상대 우위 확보**                    |
| 출장/장기 자리비움        | 8시간 1회              | 2시간 1회 + 끝나면 결과 미수령으로 시간 낭비 가능  |

**중요한 트레이드오프**: 8시간 옵션 제거는 "출근/취침 8시간 자리비움"을 1번이 아닌 4번 출발(2시간 ×4) + 중간 결과 정산을 요구. 이는 사실상 **4시간 자리비움 = 2시간 1회 끝난 뒤 미정산 = 손실**.

**완화 옵션 (별도 차후 작업)**:

- (a) 진행 중 탐험이 끝나면 자동으로 같은 지역·길이로 재출발하는 "반복 탐험" 토글
- (b) 길이 7200을 그대로 유지하되 "최대 반복 횟수"를 두어 8시간≒2시간×4를 자동화
- (c) 미정산 시 일정 시간까지는 보상 100% 보존(현재도 그러함 — 따로 손볼 필요 없음)

**본 문서 결정**: (a)~(c)는 별도 PR. 이번 작업은 버튼 3개 + 효율 곡선만.

## 8. 리스크 / 회귀 가능성

- **저레벨 진행 곡선 영향**: 1차 진행 시뮬(`scripts/first-job-progression-sim.ts`)이 어떤 길이 가정으로 작성되어 있는지 확인 필요. 60s 가정이면 사실상 변화 없음, 7200s 가정이면 EXP/골드 25% 손실로 진행 페이스 늦어짐 → 시뮬 길이를 활성 플레이 가정인 60s로 갱신.
- **사용자가 "왜 8시간 사라졌냐"** 피드백 가능성: §5 (a) 반복 탐험을 다음 PR로 약속.
