# 29. 대련 후속 메모 (백로그)

> **상태**: 메모만 — 미적용. 우선순위 낮음.
> **선행**: `427217e` 커밋 (대련 Owner token + Redis Hash 적용 완료)
> **관련 코드**: `src/app/api/arena/route.ts`, `src/lib/game/store.ts:registerArena`, `src/components/game/panels/ArenaPanel.tsx`

본 문서는 `427217e` 적용 후 남아있는 보강 아이디어를 잊지 않도록 기록만 해두는 백로그. 시점은 시급하지 않으니 나중에 필요할 때 꺼내보기 용도.

## 1. UX — 등록 토큰 사본 / 수동 입력

**왜**: localStorage 분실(시크릿 모드, 캐시 삭제, 디바이스 이동) 시 본인 닉 재등록 불가. 현재는 그냥 막힘 → 사용자는 영문도 모르고 좌절.

**무엇**:

- 등록 성공 모달에 "이 토큰을 사본으로 저장하면 다른 기기에서도 본인 등록 가능" 안내
- ownerId 수동 입력 필드 추가 (등록 폼 옵션)
- 토큰 표시 — 한 번 보여주고, 사용자가 복사할 수 있도록

**작업량**: 작음 (~30줄). ArenaPanel.tsx + 등록 응답 표시 보강.

**주의**: 토큰을 화면에 노출하므로 스크린샷/숄더 서핑 위험 일부 있음 (대련 시스템의 경우 acceptable, 결제 정보 아님).

## 2. 본격 인증 (Sign in with Vercel / Clerk)

**왜**: 멀티 디바이스 지원이 필요해질 때, 또는 본격 PvP/리더보드로 격상할 때.

**무엇**:

- Sign in with Vercel (GA 2025-11) — Vercel 생태계 통합, OAuth 흐름 단순
- 또는 Clerk (Vercel Marketplace 통합) — 더 풍부한 user 관리
- 닉네임 → user_id 매핑으로 전환 → ownerId 불필요

**작업량**: 큼 (수일~주). 인증 미들웨어 + 마이그레이션 + UI 변경 + 환경변수 설정.

**전환 트리거 (이때 검토)**:

- 멀티 디바이스 사용 사용자 비율이 높아질 때
- 닉네임 squat 클레임이 빈번해질 때
- 리더보드/시즌제/랭크 같은 영구 진행 시스템 추가 시

**비용**: Sign in with Vercel은 (현재) 무료 티어 풍부. Clerk은 MAU 기반 과금.

## 3. 그 외 작은 아이디어

- **본인 entry 자발적 삭제 API** — POST `{ action: "unregister", ownerId }` → 더 이상 도전 풀에 노출 안 됨
- **닉네임 변경 API** — 현재는 nickname 변경 시 새 entry 생성됨 (구 entry는 그대로 남음). `{ action: "rename", oldNick, newNick, ownerId }` 액션
- **풀 SCAN 비용** — 풀이 1만 명 넘어가면 hgetall 비용 부담. 페이지네이션 또는 별도 인덱스 키 추가 (현재 규모에선 무관)
- **활동 갱신** — 30일 이상 갱신 없는 entry 자동 정리 (cron) — 풀 청결도 유지

## 4. 적용 트리거 메모

- (1)은 사용자 피드백("본인 닉 갱신이 안 돼요") 1~2건 들어오면 즉시 처리 권장
- (2)는 멀티 디바이스 요구가 명확해질 때까지 미적용 권장 (가벼운 게임 정체성 유지)
- (3)은 풀 규모가 의미 있게 커지면 검토
